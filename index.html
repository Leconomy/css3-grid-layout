Abstract

This CSS module defines a two-dimensional grid-based layout system, optimized for user interface design. In the grid layout model, the children of a grid container can be positioned into arbitrary slots in a predefined flexible or fixed-size layout grid.

CSS is a language for describing the rendering of structured documents (such as HTML and XML) on screen, on paper, in speech, etc.

1. Introduction

This section is not normative.

Grid Layout is a new layout model for CSS that has powerful abilities to control the sizing and positioning of boxes and their contents. Unlike Flexible Box Layout, which is single-axis–oriented, Grid Layout is optimized for 2-dimensional layouts: those in which alignment of content is desired in both dimensions.

In addition, due to its ability to explicitly position items in the grid, Grid Layout allows dramatic transformations in visual layout structure without requiring corresponding markup changes. By combining media queries with the CSS properties that control layout of the grid container and its children, authors can adapt their layout to changes in device form factors, orientation, and available space, while preserving a more ideal semantic structuring of their content across presentations.

Although many layouts can be expressed with either Grid or Flexbox, they each have their specialties. Grid enforces 2-dimensional alignment, uses a top-down approach to layout, allows explicit overlapping of items, and has more powerful spanning capabilities. Flexbox focuses on space distribution within an axis, uses a simpler bottom-up approach to layout, can use a content-size–based line-wrapping system to control its secondary axis, and relies on the underlying markup hierarchy to build more complex layouts. It is expected that both will be valuable and complementary tools for CSS authors.

1.1. Background and Motivation

As websites evolved from simple documents into complex, interactive applications, techniques for document layout, e.g. floats, were not necessarily well suited for application layout. By using a combination of tables, JavaScript, or careful measurements on floated elements, authors discovered workarounds to achieve desired layouts. Layouts that adapted to the available space were often brittle and resulted in counter-intuitive behavior as space became constrained. As an alternative, authors of many web applications opted for a fixed layout that cannot take advantage of changes in the available rendering space on a screen.

The capabilities of grid layout address these problems. It provides a mechanism for authors to divide available space for layout into columns and rows using a set of predictable sizing behaviors. Authors can then precisely position and size the building block elements of their application into the grid areas defined by the intersections of these columns and rows. The following examples illustrate the adaptive capabilities of grid layout, and how it allows a cleaner separation of content and style.

1.1.1. Adapting Layouts to Available Space

Grid layout can be used to intelligently resize elements within a webpage. The adjacent figures represent a game with five major components in the layout: the game title, stats area, game board, score area, and control area. The author’s intent is to divide the space for the game such that:

  • The stats area always appears immediately under the game title.
  • The game board appears to the right of the stats and title.
  • The top of the game title and the game board should always align.
  • The bottom of the game board and bottom of the stats area align when the game has reached its minimum height. In all other cases the game board will stretch to take advantage of all the space available to it.
  • The controls are centered under the game board.
  • The top of the score area is aligned to the top of the controls area.
  • The score area is beneath the stats area.
  • The score area is aligned to the controls beneath the stats area.

The following grid layout example shows how an author might achieve all the sizing, placement, and alignment rules declaratively.

/**
 * Define the space for each grid item by declaring the grid
 * on the grid container.
 */
#grid {
  /**
   * Two columns:
   *  1. the first sized to content,
   *  2. the second receives the remaining space
   *     (but is never smaller than the minimum size of the board
   *     or the game controls, which occupy this column [Figure 4])
   *
   * Three rows:
   *  3. the first sized to content,
   *  4. the middle row receives the remaining space
   *     (but is never smaller than the minimum height
   *      of the board or stats areas)
   *  5. the last sized to content.
   */
  display: grid;
  grid-template-columns:
    /* 1 */ auto
    /* 2 */ 1fr;
  grid-template-rows:
    /* 3 */ auto
    /* 4 */ 1fr
    /* 5 */ auto;
}

/* Specify the position of each grid item using coordinates on
 * the 'grid-row' and 'grid-column' properties of each grid item.
 */
#title    { grid-column: 1; grid-row: 1; }
#score    { grid-column: 1; grid-row: 3; }
#stats    { grid-column: 1; grid-row: 2; align-self: start; }
#board    { grid-column: 2; grid-row: 1 / span 2; }
#controls { grid-column: 2; grid-row: 3; justify-self: center; }

<div id="grid">
  <div id="title">Game Title</div>
  <div id="score">Score</div>
  <div id="stats">Stats</div>
  <div id="board">Board</div>
  <div id="controls">Controls</div>
</div>

Note: There are multiple ways to specify the structure of the grid and to position and size grid items, each optimized for different scenarios.

1.1.2. Source-Order Independence

Continuing the prior example, the author also wants the game to adapt to different devices. Also, the game should optimize the placement of the components when viewed either in portrait or landscape orientation (Figures 6 and 7). By combining grid layout with media queries, the author is able to use the same semantic markup, but rearrange the layout of elements independent of their source order, to achieve the desired layout in both orientations.

The following example uses grid layout’s ability to name the space which will be occupied by a grid item. This allows the author to avoid rewriting rules for grid items as the grid’s definition changes.

@media (orientation: portrait) {
  #grid {
    display: grid;

    /* The rows, columns and areas of the grid are defined visually
     * using the grid-template-areas property.  Each string is a row,
     * and each word an area.  The number of words in a string
     * determines the number of columns. Note the number of words
     * in each string must be identical. */
    grid-template-areas: "title stats"
                         "score stats"
                         "board board"
                         "ctrls ctrls";

    /* The way to size columns and rows can be assigned with the
     * grid-template-columns and grid-template-rows properties. */
    grid-template-columns: auto 1fr;
    grid-template-rows: auto auto 1fr auto;
  }
}

@media (orientation: landscape) {
  #grid {
    display: grid;

    /* Again the template property defines areas of the same name,
     * but this time positioned differently to better suit a
     * landscape orientation. */
    grid-template-areas: "title board"
                         "stats board"
                         "score ctrls";

    grid-template-columns: auto 1fr;
    grid-template-rows: auto 1fr auto;
  }
}

/* The grid-area property places a grid item into a named
 * area of the grid. */
#title    { grid-area: title }
#score    { grid-area: score }
#stats    { grid-area: stats }
#board    { grid-area: board }
#controls { grid-area: ctrls }

<div id="grid">
  <div id="title">Game Title</div>
  <div id="score">Score</div>
  <div id="stats">Stats</div>
  <div id="board">Board</div>
  <div id="controls">Controls</div>
</div>

Note: The reordering capabilities of grid layout intentionally affect only the visual rendering, leaving speech order and navigation based on the source order. This allows authors to manipulate the visual presentation while leaving the source order intact and optimized for non-CSS UAs and for linear models such as speech and sequential navigation.

Grid item placement and reordering must not be used as a substitute for correct source ordering, as that can ruin the accessibility of the document.

2. 概览

本部分不具有规范性

网格布局通过使用 grid 来控制其内容的布局. grid 是一组相交的水平线和垂直线。这两条线为 grid container的内容创建一个尺寸和定位坐标系. 网格布局功能

  • 固定的，弹性的和基于内容的track sizing functions
  • 通过向前（正向）和向后（负向）数值网格坐标定义的 explicit item placement 被命名为网格线和网格区域; 自动项目放置到空白区域, 包括按 order属性重新排序
  • 空间敏感追踪重复 and 自动添加额外的行和列以容纳额外的内容
  • 控制对齐 和 使用 margins, gutters进行间隔, 以及 对齐属性 alignment properties
  • 可以内容层叠 和使用 z-index控制分层

Grid containers 可以根据需要进行嵌套或者和 flex containers 一起混合以创建更复杂的布局

2.1. Grid的声明

网格轨道 tracks (即rows and columns) 可以通过显示网格属性 explicit grid 显示声明和定义大小，也可以当项目放置在显示网格 explicit grid外时进行隐式创建. grid 的简写和它的子属性定义了grid的参数. §7 Defining the Grid

以下是网格声明的一些例子:
  • 以下是具有 H, A, B, 和 F四个命名区域声明的网格. 第一列内容自适应 (auto), 第二列占据剩余空间 (1fr). 行默认为内容自适应 auto (content-based); 最后一行设置为 30px的固定尺寸. 
    main {
  grid: "H    H "
        "A    B "
        "F    F " 30px
  /     auto 1fr;
}
  • 以下声明的网格具有尽可能多的行数，每行最小5em ，以适应网格容器的高度 (100vh). 网格没有显示的列; 而是在添加内容时添加列, 使得列的宽度相等且为 (1fr). 由于向右溢出的内容不会打印，因此打印的替代布局会添加行. 
    main {
  grid: repeat(auto-fill, 5em) / auto-flow 1fr;
  height: 100vh;
}
@media print {
  main {
    grid: auto-flow 1fr / repeat(auto-fill, 5em);
  }
}
  • 以下声明一个包含5个大小均匀的列和三行的网格，中间的一行占用所有剩余空间（并且至少足以容纳其内容）. 
    main {
  grid: auto 1fr auto / repeat(5, 1fr);
  min-height: 100vh;
}

2.2. 布局项目

网格容器 grid container的内容被组织成单独的网格项目grid items（类似于弹性项目flex items）, 然后将其分配给网格 grid中的预定义区域 areas. 既可以通过网格布局属性 grid-placement properties使用坐标显示放置它们 也可以使用自动布局 auto-placement隐式放置到空白区域. §8 Placing Grid Items

以下是使用网格区域 grid-area进行网格布局声明的一些示例: 
grid-area: a;          /* Place into named grid area “a”     */
grid-area: auto;       /* Auto-place into next empty area    */
grid-area: 2 / 4;      /* Place into row 2, column 4         */
grid-area: 1 / 3 / -1; /* Place into column 3, span all rows */
grid-area: header-start / sidebar-start / footer-end / sidebar-start;
                       /* Place using named lines            */

相当于使用 grid-row + grid-column 进行如下的声明:

grid-row: a;                         grid-column: a;
grid-row: auto;                      grid-column: auto;
grid-row: 2;                         grid-column: 4;
grid-row: 1 / -1;                    grid-column: 3;
grid-row: header-start / footer-end; grid-column: sidebar-start / footer-end;

他们可以进一步被分解为 grid-row-start/grid-row-end/grid-column-start/grid-column-end 等等

grid-area: a;
/* Equivalent to grid-row-start: a; grid-column-start: a; grid-row-end: a; grid-column-end: a; */

grid-area: 1 / 3 / -1;
/* Equivalent to grid-row-start: 1; grid-column-start: 3; grid-row-end: -1; grid-column-end: auto; */

2.3. 调整网格大小

一旦网格项目 grid items 被定位 placed, 网格轨道 grid tracks (rows and columns) 就会被计算, 并根据网格定义中指定的内容大小和/或可用空间.

根据网格容器 grid container的align-content 和 justify-content属性，在生成的网格容器内对齐aligned §10 Alignment and Spacing

下面的例子是通过分配额外的空间对齐所有列,如果小于100vh则在网格容器 grid container内对齐网格. 
main {
  grid: auto-flow 1fr / repeat(auto-fill, 5em);
  min-height: 100vh;
  justify-content: space-between;
  align-content: safe center;
}

最后，按照自己被设定的尺寸sizing [CSS21]和对齐属性alignment properties [CSS-ALIGN-3]，每一个网格项grid item 都在其被分配的网格区域grid area内调整尺寸和对齐方式.

3. 网格布局的概念和术语

在 网格布局中, 网格容器grid container 的内容通过定位和对齐到网格中进行布局的. 网格 grid是一组相交的水平和垂直网格线grid lines，将网格容器grid container的空间划分为网格区域grid areas，划分为可以放置内容的网格项grid items（代表网格容器的内容）. 有两组网格线 grid lines: 一组定义沿着竖轴运行的列 columns (即列轴 column axis), 另一组是沿着横轴inline axis运行的行 rows(即行轴 row axis). [CSS3-WRITING-MODES]

Note: The “column axis” term as used here was chosen to match the block axis because this matches the visual orientation of the column boxes themselves and matches the direction that a column grows as more items are added to it. As a result, it does not match the axis in which new columns are added or in which their track size is measured, which is the inline or row axis The opposite logic, of course, applies to the “row axis”.

3.1. 网格线

网格线 Grid lines 是 grid垂直和水平方向的分界线.一条网格线grid line 存在行或列的任一侧. 它们可以通过数字索引或作者指定的名称来引用. 网格项grid item 引用网格线 grid lines 以使用网格布局属性grid-placement properties确定其在网格grid内的位置.

以下两个例子都是创建三条列网格线 column grid lines和 四条行网格线 row grid lines.

第一个示例演示了作者如何使用网格线号grid line定位网格项目 grid item:

#grid {
  display: grid;
  grid-template-columns: 150px 1fr;
  grid-template-rows: 50px 1fr 50px;
}

#item1 { grid-column: 2;
         grid-row-start: 1; grid-row-end: 4; }

第二个示例使用显式命名的网格线 grid lines:

/* equivalent layout to the prior example, but using named lines */
#grid {
  display: grid;
  grid-template-columns: 150px [item1-start] 1fr [item1-end];
  grid-template-rows: [item1-start] 50px 1fr 50px [item1-end];
}

#item1 {
  grid-column: item1-start / item1-end;
  grid-row: item1-start / item1-end;
}

3.2. 网格轨道和单元格

网格轨道 Grid track 是 网格列grid column 和 网格行grid row的通用叫法-换句话说, 它是两条相邻的网格线 grid lines之间的空间. 每一个网格轨道 grid track 都被分配一个调整大小的函数, 这个调整大小的函数控制这列或行可能增长的宽度或者高度, 即到它边界的网格线 grid lines 的距离. 相邻的网格轨道 grid tracks 会被 gutters分开 否则它们会紧紧贴在一起.

A 网格单元格 grid cell 是网格行和网格列的交点. 它是定位网格项目 grid items是可以引用的网格最小单位.

下面是一个有三行两列的例子. 第一列固宽150px. 第二列使用自适应布局, 它会随着网格容器 grid container 的宽度变化而变化. 如果网格容器 grid container 宽度是200px, 第二列的宽度便是50px. 如果网格容器 grid container 宽度是 100px, 那么第二列的宽度便是0px 同时任何内容被放置在此列都会溢出网格容器 grid container. 
#grid {
  display: grid;
  grid-template-columns: 150px 1fr;  /* two columns */
  grid-template-rows: 50px 1fr 50px; /* three rows  */
}

3.3. 网格区域

网格区域 grid area 是用来布置一个或多个网格项目 grid items的逻辑空间. 网格区域grid area 包含一个或多个网格单元格 grid cells. 它受四条网格线 grid lines 限制, 网格区域 grid area 的每一侧都有一条网格线, 并且参与其相交的网格轨道 grid tracks 的大小. 网格区域 grid area 可以使用网格容器grid container的网格模版区域grid-template-areas属性进行显示命名 或由其边界网格线 grid lines隐式引用. 网格项 grid item 使用网格布局grid-placement properties属性分配给网格区域 grid area.

/* using the template syntax */
#grid  {
display: grid;
grid-template-areas: ". a"
                     "b a"
                     ". a";
grid-template-columns: 150px 1fr;
grid-template-rows: 50px 1fr 50px;
}

#item1 { grid-area: a }
#item2 { grid-area: b }
#item3 { grid-area: b }

/* Align items 2 and 3 at different points in the grid area "b".  */
/* By default, grid items are stretched to fit their grid area    */
/* and these items would layer one over the other. */
#item2 { align-self: start; }
#item3 { justify-self: end; align-self: end; }

网格项 grid item的网格区域 grid area 形成了它所包含的块区域. 放置在同一网格区域grid area中的网格项Grid items 不会直接影响彼此的布局. 但是, 具有固定尺寸的占据网格轨道grid track的网格项grid item会间接影响该网格轨道的尺寸(并且影响其边界网格线grid lines的位置), 这反过来又影响另一个网格项grid item的位置和尺寸.

4. 重新排序和可访问性

作者可以很方便的使用网格布局对文档进行重排. 但是，这些不能替代文档来源的正确排序. 在非视觉媒介如(speech)中, 排序order 属性和网格布局 grid placement属性不能影响排序. 同样, 重新排列可视化的网格项也不会影响顺序导航模式 (如链接循环,请参考 tabindex [HTML5])的默认遍历顺序.

作者必须使用 order 和 grid-placement 属性 仅用于可视而不是逻辑内容的排序. 使用这些功能执行逻辑重新排序的样式表不符合要求.

注意: 这样，非可视媒体和非CSS UA（通常呈现线性内容）可以依赖逻辑源顺序，而网格布局的布局和排序功能用于调整可视布局。 （由于视觉感知是二维的和非线性的，所需的视觉顺序并不总是等同于所需的阅读顺序。）

很多网页都有相似的解构, 一个header在顶, 一个footer在底, 包含一列或两列的内容区在中间. 通常,我们希望内容在附加列之前显示.但是,这使得需要常见设计难以实现如左侧的附加列和右侧的内容区有. 多年来这已经有了多种解决方案, 当有两个额外的栏时，通常以“圣杯布局”的名字命名. 网格布局使这个例子变得微不足道，很容易实现. 例如下面的代码和草图: 
<!DOCTYPE html>
<header>...</header>
<article>...</article>
<nav>...</nav>
<aside>...</aside>
<footer>...</footer>

上面的布局使用网格布局可以很容器实现:

main { display: grid;
       grid: "h h h"
             "a b c"
             "f f f";
       grid-template-columns: auto 1fr 20%; }
article { grid-area: b; min-width: 12em;     }
nav     { grid-area: a; /* auto min-width */ }
aside   { grid-area: c; min-width: 12em;     }

有个好处是, 列默认是等高 equal-height的, 而且主内容会自适应屏幕. 此外可以结合媒体查询在窄屏上进行垂直布局:

@media all and (max-width: 60em) {
  /* Too narrow to support three columns */
  main { display: block; }
}

为了保持作者在所有表示模式下的预定顺序, 工具包括 WYSIWYG 编辑器 以及基于web的编辑器— 必须对底层文档进行排序 不要使用 order 或 grid-placement properties 进行重新排序，除非作者有明确的指出底层文档的顺序 (确定语音和导航顺序) 应该与视觉顺序同步.

For example, a tool might offer both drag-and-drop arrangement of grid items as well as handling of media queries for alternate layouts per screen size range.

Since most of the time, reordering should affect all screen ranges as well as navigation and speech order, the tool would match the resulting drag-and-drop visual arrangement by simultaneously reordering the DOM layer. In some cases, however, the author may want different visual arrangements per screen size. The tool could offer this functionality by using the grid-placement properties together with media queries, but also tie the smallest screen size’s arrangement to the underlying DOM order (since this is most likely to be a logical linear presentation order) while using grid-placement properties to rearrange the visual presentation in other size ranges.

This tool would be conformant, whereas a tool that only ever used the grid-placement properties to handle drag-and-drop grid rearrangement (however convenient it might be to implement it that way) would be non-conformant.

不严格的说, 网格容器grid container的网格项 grid items 是代表流入其内容的盒子.

一个网格项 grid item 为它的内容创建一个新的格式上下文. 此格式上下文的类型取决于 display 的值. 但是, 网格项是网格级 grid-level 盒模型, 不是块级盒模型: 他们参与父容器的网格格式上下文 grid formatting context不在块级格式上下文中.

注意: display 的某些值通常会在原始盒模型的周围触发创建匿名盒模型. 如果盒模型是网格项 grid item, 它首先被阻塞, 并且匿名盒模型也不会被创建. 例如, 两个具有display: table-cell属性的相邻的 grid items 将会被分隔为具有 display: block属性的两个 grid items, 而不是被包装成一个单独的匿名table.

7. 定义网格

7.1. 显式网格

grid-template-rows, grid-template-columns, 和 grid-template-areas 这三个属性一起定义了网格容器 grid container的显式网格 explicit grid. 由于网格项grid items放置在显式网格explicit grid之外，最终的网格可能会变大; 在这种情况下，隐式轨道将会被创建, 这些隐式轨道大小将有 grid-auto-rows 和 grid-auto-columns 属性决定.

显式网格 explicit grid 的大小由grid-template-areas 定义的行或列数中较大的一个决定以及 grid-template-rows/grid-template-columns大小确定的行数或列数决定. 使用 grid-template-areas 定义而没有使用 grid-template-rows/grid-template-columns 定义的任何行或列，它们会自动从 grid-auto-rows/grid-auto-columns 属性获取大小. 如果这些属性没有定义任何明确的轨道，则显式网格explicit grid在每个轴中仍然包含一条网格线grid line.

grid 和 grid-template 属性是一个简写属性shorthands, 它可以同时设置三个显式网格属性 explicit grid properties (grid-template-rows, grid-template-columns, 和 grid-template-areas). grid 也可以重置控制隐式网格属性 implicit grid, 而 grid-template 属性会保持其不变.

7.2.2. 重复行和列: repeat()

repeat() 展现一个重复的轨道列表 track list片段, 允许大量显示重复模式的列或行以更紧凑的形式写入.

这个例子展示了两种书写同一个网格的等价方法. 每个都产出四个主列, 每列250px宽, 列直接有10px间隔. 
grid-template-columns: 10px [col-start] 250px [col-end]
                       10px [col-start] 250px [col-end]
                       10px [col-start] 250px [col-end]
                       10px [col-start] 250px [col-end] 10px;
/* same as above, except easier to write */
grid-template-columns: repeat(4, 10px [col-start] 250px [col-end]) 10px;
7.2.2.1. repeat()的语法

repeat() 语法的通用形式近似于,

repeat( [ <positive-integer> | auto-fill | auto-fit ] , <track-list> )

第一个参数指定了重复的次数. 第二个参数是重复该次数的轨道列表 track list. 然后，这有些限制:

  • repeat()不能嵌套.

  • 自动重复 (auto-fill or auto-fit) 不能和固定尺寸 intrinsic 或 灵活flexible 尺寸联合使用.

因此 repeat()的精确语法有如下几种方式:

<track-repeat> = repeat( [ <positive-integer> ] , [ <line-names>? <track-size> ]+ <line-names>? )
<auto-repeat>  = repeat( [ auto-fill | auto-fit ] , [ <line-names>? <fixed-size> ]+ <line-names>? )
<fixed-repeat> = repeat( [ <positive-integer> ] , [ <line-names>? <fixed-size> ]+ <line-names>? )

  • <track-repeat> 可以表示任何 <track-size>的重复, 但有固定次数的限制.

  • <auto-repeat> 可以自动重复填充空间, 但需要固定的 definite 轨道大小以便计算重复次数. 它只能在轨道列表 track list中出现一次, 但相同的轨道列表 track list 也可以包含多个 <fixed-repeat>.

如果 repeat() 函数最终将两条相邻的网格线放置在一起<line-names>, 那么它们会合并. 例如, repeat(2, [a] 1fr [b]) 等价于 [a] 1fr [b a] 1fr [b].

7.2.2.2. 重复填充: auto-fill 和 auto-fit

当重复数字是 auto-fill 时, 如果网格容器 grid container 有固定尺寸 definite 或者最大尺寸, 那此重复数字是可能的最大正数值,它不会导致网格 grid 溢出网格容器 grid container (将每条轨道视为最大轨道尺寸函数 max track sizing function 如果此轨道函数是固定的 definite 或者最小尺寸函数,否则就会考虑计算间距 gap); 如果任何的重复数字溢出了, 那只会重复一次. 否则, 如果网格容器 grid container 有固定 definite 最小尺寸函数, 则重复次数是满足该最小尺寸要求的最小可能正整数. 否则, 指定的轨道列表 track list 只会重复一次.

例如，下面的代码将创建25ch的列，以适应窗口宽度。如果有剩余空间，它将分布在25ch的列中. 
body {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(25ch, 1fr));
}

auto-fit 表现如同 auto-fill, 只不过它在网格布局 grid item placement 之后,任何空的重复的轨道都会被折叠 collapsed. 空轨道是没有放入或跨越其中的流入网格物体的轨道. (这可能导致所有轨道被折叠 collapsed, 如果他们全是空的.)

折叠轨道 collapsed track 被视为具有0px的固定轨道尺寸函数 track sizing function,并且两边都有间距 gutters 包括通过分布式对齐分配的任何控件 distributed alignment— collapse.

为了找到自动重复轨道的数量，UA必须将轨道大小设置为UA指定的向下取整的值以避免被零除。建议这个向下取整的值是 1px.

7.2.3. 灵活长度值: fr

灵活长度 或者 <flex> 是以 fr 为单位的尺寸, 它标识网格容器grid container剩余空间 leftover space的一部分 . 以 fr 为单位的轨道被称为灵活轨道 flexible tracks因为它们能灵活的响应剩余空间 leftover space 类似于 flex items 如何填充 flex container.

注意: <flex> 值不是 <length>s (也不兼容 <length>s, 比如一些百分比 <percentage> values), 因此它们不能在calc()calc()表达式中表示或与其他单位类型结合使用.

在所有非灵活轨道尺寸达到了他们的最大值后，将产生剩余空间 leftover space. 将行或列的总大小从可用空间中扣除, 便产生剩余空间 leftover space, 然后根据他们的弹性系数按比例在灵活大小的行或列里进行分隔 flex factor.

注意: 轨道列表 track list中的灵活长度与[CSS-FLEXBOX-1]中的基本大小为零的柔性长度类似.

每行或列所占有的剩余空间 leftover space 可以计算为每行或列的 <flex> * <leftover space> / <sum of all flex factors>.

注意: 如果弹性系数 flex factors的总和小于1，它们将只占用剩余空间 leftover space的相应部分，而不是扩大以填充整个事物. 这与Flexbox [CSS-FLEXBOX-1] 在 flex值总和小于1时的行为方式类似.

当可用空间无限大时 (即当网格容器 grid container的宽和高无限大 indefinite), 弹性网格轨道grid tracks的大小根据其内容而保持其各自的比例. 通过确定每个弹性尺寸网格轨道grid track的最大内容max-content尺寸并将该尺寸除以相应的弹性系数以确定"假定1fr尺寸"来计算每个弹性尺寸网格轨道的使用尺寸. 这些最大值用作解析后的1fr长度（弹性部分），然后将其乘以每个网格曲线的弹性因子flex factor以确定其最终大小.

8.1. 网格布局的常见模式

本节内容只是告知性的.

网格布局属性grid-placement property 的普通写法可以拆分为三种简写写法:

grid-area
grid-column grid-row
grid-column-start grid-column-end grid-row-start grid-row-end

8.1.1. 命名区域

通过在网格区域grid-area中指定区域名,可以把一个网格项定位进命名网格区域中 named grid area (如在 grid-template-areas中由模版生成的那些):

article {
  grid-area: main;
  /* Places item into the named area "main". */
}

项目也可以与命名网格区域named grid area部分对齐，其他边缘与其他网格线对齐:

.one {
  grid-row-start: main;
  /* Align the row-start edge to the start edge of the "main" named area. */
}

8.1.2. 数字索引和跨度

网格项目可以通过数字进行定位和大小，这对脚本驱动的布局特别有用:

.two {
  grid-row: 2;    /* Place item in the second row. */
  grid-column: 3; /* Place item in the third column. */
  /* Equivalent to grid-area: 2 / 3; */
}

默认情况下，网格项的跨度为1.可以明确给出不同的跨度:

.three {
  grid-row: 2 / span 5;
  /* Starts in the 2nd row,
     spans 5 rows down (ending in the 7th row). */
}

.four {
  grid-row: span 5 / 7;
  /* Ends in the 7th row,
     spans 5 rows up (starting in the 2nd row). */
}

注意: 网格索引是相对书写模式 writing mode. 例如, 在从右向左的书写语言如阿拉伯语言中, 第一列就是最右边的列.

8.1.3. 命名网格线和网格跨度

可以通过名称来引用命名网格线named lines, 而不是按数字对网格线进行计数:

.five {
  grid-column: first / middle;
  /* Span from line "first" to line "middle". */
}

注意: 如果命名网格区域 named grid area 和命名网格线 named line 有相同的名字, 那布局算法将倾向于使用命名网格区域 named grid area的线.

If there are multiple lines of the same name, they effectively establish a named set of grid lines, which can be exclusively indexed by filtering the placement by name:

.six {
  grid-row: text 5 / text 7;
  /* Span between the 5th and 7th lines named "text". */
  grid-row: text 5 / span text 2;
  /* Same as above - start at the 5th line named "text",
     then span across two more "text" lines, to the 7th. */
}

8.1.4. 自动布局

网格项grid item 可以自动定位到下一个可用的空网格单元 grid cell中, 如果没有剩余空间这生成网格 grid.

.eight {
  grid-area: auto; /* Initial value */
}

例如，这可用于以网格模式列出目录网站上的许多销售物品.

如果网格项应占用多个单元格，自动布局可以与显式的跨度结合使用:

.nine {
  grid-area: span 2 / span 3;
  /* Auto-placed item, covering two rows and three columns. */
}

自动布局算法 auto-placement algorithm是搜索并添加行，还是搜索并添加列，都由 grid-auto-flow属性控制.

注意: 默认情况下，自动布局算法auto-placement algorithm在没有回溯的情况下通过网格线性查找; 如果它不得不跳过一些空的空间来放置更大的网格项, 它不会返回来填补这些空间. 要更改此行为, 请设置grid-auto-flow属性值为 dense关键字.

8.2. 网格项的布局 vs. 源顺序

“拥有权利的同时也被赋予了重大的责任.”

网格布局属性grid-placement properties的能力允许内容在网格grid内自由排列和重新排序，使得视觉呈现可以基本上与底层文档源顺序不相交. 这些能力允许开发者在渲染不同设备和呈现模式有很大的自由 比如使用 media queries. 但是，它们不能替代正确的源排序.

对演讲、顺序导航(例如键盘导航)以及无CSS的UA如搜索引擎触觉浏览器等, 正确的源顺序是很重要的. 网格布局仅影响视觉呈现! 这允许开发者优化非CSS /非视觉交互模式的文档源，并使用网格布局技术来进一步操纵视觉呈现，以便完整保留源顺序.

8.5. 网格项布局算法

以下 网格项布局算法 解析 网格项grid items的自动定位automatic positions到指定位置definite positions, 确保每一个网格项 grid item 都能布局进一个定义好的网格区域 grid area. ( Grid spans 不需要特殊的解决方案; 如果它们没被显式指定则默认为1.)

注意: 如果显式网格中没有空间放置自动定位的网格项，则此算法可能会导致在隐式网格中创建新的行或列.

每个单元格 grid cell (包括显式 explicit 和 隐式implicit grids) 都可以被占用 或者不占用. 如果一个单元格被具有确定网格位置的网格项目的网格区域覆盖，则该单元格被占用;否则就没被占用. 在算法期间,单元格是否占用的状态是可以被改变的.

为了清楚起见, 此算法是在假定 grid-auto-flow 有指定 row 下编写的. 在此算法中, 如果它被列替换 column, 则交互所有的列和行,内联和块级等.

注意: 自动布局算法 auto-placement algorithm 是作用在视觉层的网格项 grid items, 而不是作用在底层顺序.

  1. 生成匿名网格项 如同在 §6 Grid Items描述一样. (匿名网格项 grid items 总是自动布局, 因为他们的盒子不可能具有指定的网格布局属性grid-placement properties.)

  2. 定位任何不能自动定位的东西.

  3. 处理锁定到给定行的项目.

9. 绝对定位

9.1. 以网格容器作为包含块

如果一个绝对定位元素的包含块 containing block 是由网格容器 grid container生成, 则包含块对应于由其网格布局属性确定的网格区域. 正常情况下，偏移属性(top/right/bottom/left)指示从该包含块的相应边向内偏移.

注意:虽然将元素绝对定位到网格容器确实允许它与该容器的网格线对齐，但这些元素不占用空间或以其他方式参与网格布局.

.grid {
  grid: 1fr 1fr 1fr 1fr / 10rem 10rem 10rem 10rem;
  /* 4 equal-height rows filling the grid container,
     4 columns of 10rem each */
  justify-content: center;
  /* center the grid horizontally within the grid container */
  position: relative;
  /* Establish abspos containing block */
}

.abspos {
  grid-row-start: 1;     /* 1st grid row line = top of grid container */
  grid-row-end: span 2;  /* 3rd grid row line */
  grid-column-start: 3;  /* 3rd grid col line */
  grid-column-end: auto; /* right padding edge */
  /* Containing block covers the top right quadrant of the grid container */

  position: absolute;
  top: 70px;
  bottom: 40px;
  left: 100px;
  right: 30px;
}

注意: 网格和网格布局属性 grid-placement properties 都是相对流 flow-relative, 但偏移属性 (left, right, top, and bottom) 却是物理性的 physical, 所以如果 direction 或者 writing-mode 属性改变, 网格也会做相应的改变, 但是偏移属性不会.

网格布局属性的auto值不是自动布局，而是为位置与网格容器的相应填充边缘位置相对应的布局提供特殊线条 (如果网格容器溢出, 可滚动区域的填充边缘). T这些线条成为用于定位绝对定位项目的增广网格的第一行和最后一行(0th and -0th).

注意: 因此，默认情况下，绝对定位的框的包含块将对应于网格容器的填充边缘，就像它对块容器所做的那样.

绝对定位发生在网格布局和其流入内容之后，并且不会影响任何网格轨道的大小或以任何方式影响网格的大小/配置. 如果网格布局属性通过显式指定这样一条线或跨越现有的隐式网格引用不存在的线，则它将被视为指定auto (而不是创建新的隐式网格线).

如果布局仅包含grid span，请将其替换为该轴中的两条auto线. (当轴中的网格布局属性最初贡献了一个跨度时会发生这种情况，并且第8.3.1节“网格布局冲突处理”导致第二个跨度被忽略.)

9.2. 以Grid Container作为父容器

网格容器grid container的绝对定位的子节点不在文档流中，也不是grid item，因此不会影响其他项目的布局或网格大小.

网格容器的绝对定位的子节点的static position[CSS21]被确定为它是网格区域中唯一的网格物体，其边缘与网格容器的padding边缘重合. 但是，如果网格容器父容器也是绝对定位元素的包含块的生成器，请改为使用§9.1规则.

注意: 请注意，此定位受子元素上的justify-self和align-self的值影响，并且与大多数其他布局模型一样，绝对定位的子元素对包含的块的大小或其内容的布局没有影响内容.

10. 对齐和间距

在网格容器grid container的网格轨道grid tracks确定尺寸并且所有网格项grid items也确定尺寸后，网格项可以在其网格区域grid areas内对齐.

margin 属性可以用于对齐网格项如同margin在块级布局中所起作用一样. 网格项Grid items还考虑CSS Box Alignment ModuleCSS Box Alignment Module [CSS-ALIGN-3][CSS-ALIGN-3]中的盒子对齐属性box alignment properties，它允许在行轴和列轴上轻松实现基于关键字的项目对齐.

默认情况下，网格项目拉伸以填充其网格区域. 但是，如果justify-self或align-self计算为非stretch或margin的值为auto，则网格项目将自动调整大小以适应其内容.

10.1. 间距: row-gap, column-gap, 和 gap 属性

当在网格容器grid container上定义 row-gap 和 column-gap 属性 (及gap 缩写)时, 便定义了grid rows 和 grid columns之间的间距 gutters. 它们的语法请详见 CSS Box Alignment 3 §8 Gaps Between Boxes.

T这些属性的效果好像受影响的网格线获得了厚度: 两条网格线之间的网格轨道是代表它们的间距之间的空间. 为了计算轨道尺寸track sizing，每个间距gutter都被视为指定尺寸的额外空轨道.

Note: 由于justify-content/align-content的原因,可能会在轨道直接添加额外的间距. 请参见 §11.1 网格尺寸算法. 这个间距有效的增加 gutters 的大小.

如果网格 grid 在轨道直接是碎片 fragmented, 则必须压缩这些轨道之间的间距gutter. 请注意，即使在强制中断后，gugger也会受到抑制，这与margins不同.

Gutters 仅出现在隐式网格implicit grid轨道之间; 在第一个轨道之前或者最后一个轨道之后是没有gutters的. (特别是，隐式网格的第一条/最后一条轨道与augmented grid中的“自动”线之间没有间隙.)

当一个折叠轨道collapsed track的 gutters 折叠时, 他们完全吻合— 两条 gutters 重叠以便他们的开始边和结合边重合. 如果折叠轨道的一边没有 gutter (比如 如果它是隐式网格的第一个或者最后一个轨道), 那么折叠它的gutter导致在折叠轨道collapsed track的另一边也没有gutter.

10.2. 使用auto margin对齐

此节内容不是规范. 关于margin如何影响网格项的规范定义在§11 Grid Sizing中.

网格项grid items的Auto margins与块级流的auto margins非常相似:

  • 在计算网格轨道大小过程中, auto margins被视为 0.
  • auto margins 在通过box alignment properties进行对齐之前会充分利用可用空间.
  • 溢出元素会忽略 auto margins 并且有他们的box alignment properties指定溢出.

10.3. 行轴对齐: justify-self 和 justify-items 属性

按照[CSS-ALIGN-3]中的定义，通过在网格项目上使用justify-self属性或在网格容器上使用justify-items属性，可以在内联维度中对齐网格项目.

例如，对于英文文档，内联轴是水平的，因此justify- *属性水平对齐网格项.

如果在网格项上指定了基线对齐baseline alignment，该网格项的大小取决于内在大小的轨道（因此其大小取决于项目的大小和基线对齐，创建循环依赖项）的大小，该项目不参与 在基线对齐中，而是使用其后退对齐方式 fallback alignment.

10.4. 列轴对齐: align-self 和 align-items 属性

按照[CSS-ALIGN-3]中所定义，通过在网格项上使用align-self属性或在网格容器上使用align-items属性，网格项也可以在块维度上（垂直于内联维度）对齐.

如果在网格项上指定了基线对齐baseline alignment，该网格项的大小取决于内在大小的轨道（因此其大小取决于项目的大小和基线对齐，创建循环依赖项）的大小，该项目不参与 在基线对齐中，而是使用其后退对齐方式 fallback alignment.

10.5. 对齐网格: justify-content 和 align-content 属性

如果网格的外边缘不对应网格容器的内容边缘 (例如，如果没有列是弹性大小的), 网格轨道 grid tracks 根据网格容器上的justify-content 和 align-content属性在内容框中对齐.

例如，以下网格垂直居中，并与其网格容器的右边缘对齐: 
.grid {
  display: grid;
  grid: 12rem 12rem 12rem 12rem / 10rem 10rem 10rem 10rem;
  justify-content: end;
  align-content: center;
}

如果没有网格轨道 grid tracks（显式网格为空explicit grid，并且隐式网格implicit grid中未创建轨道），则每个轴中唯一的网格线与网格容器的起始边缘对齐.

请注意 justify-content 和 align-content 的某些值可能会导致轨道间隔开 ( space-around, space-between, space-evenly)或重新调整大小 (stretch). 如果网格在轨道之间被分割，则必须抑制这些轨道之间的任何额外空间.

例如，在下面的网格中，跨度项目的网格面积会增加以适应由于对齐而分配给gutter的额外空间: 
.wrapper {
  display: grid;
  /* 3-row / 4-column grid container */
  grid: repeat(3, auto) / repeat(4, auto);
  gap: 10px;
  align-content: space-around;
  justify-content: space-between;
}

.item1 { grid-column: 1 / 5; }
.item2 { grid-column: 1 / 3; grid-row: 2 / 4; }
.item3 { grid-column: 3 / 5; }
/* last two items auto-place into the last two grid cells */

请注意，对齐（与gap间距不同）发生在网格轨道大小调整后，所以如果轨道大小由跨越项目的内容决定，那它将在对齐阶段获得多余空间以适应对齐间距.

10.6. 网格容器基线

网格容器的第一个（最后一个）基线确定如下:

  1. 如果其区域与网格容器grid container的第一行（最后一行）相交的任何网格项参与基线对齐baseline alignment，则网格容器的基线组baseline set将从这些网格项的共享对齐基线生成.
  2. 否则，如果网格容器至少有一个网格项目的区域与第一个（最后一个）行相交，则网格容器的第一个（最后一个）基准集合是从第一个（最后一个）这样的网格项目的对齐基线生成的(在主行网格顺序中). 如果该项目在网格的内联轴上没有对齐基线alignment baseline，则首先从其边界边缘合成synthesized一个.
  3. 否则，网格容器没有第一个（最后一个）基线集合baseline set，根据其对齐上下文alignment context的规则合成一个.

Grid-modified document order (grid order) 是在遍历网格的网格单元格时遇到网格项的顺序. 如果同时遇到两个项目, 他们将按 order-modified document order顺序排列.

根据上述规则计算基线时，如果提供基线的盒子具有允许滚动的溢出overflow值，则必须将该盒子视为处于其初始滚动位置以确定其基线.

在确定表格单元格的基线时，网格容器就像线框或表格行一样提供基线. [CSS21]

详见 CSS Writing Modes 3 §4.1 Introduction to Baselines 和 CSS Box Alignment 3 §9 Baseline Alignment Details 章节,获取有关基线的更多信息.

11. 网格尺寸

本部分定义了网格大小调整算法，该算法确定所有网格轨道的大小以及整个网格的大小.

每条轨道都指定了 最小 和 最大 的sizing functions (这可能是相同的). 每个 sizing function 都是:

  • 一个固定的尺寸调整功能 ( <length>或 可用的 <percentage>).
  • 内在的尺寸调整功能 (min-content, max-content, auto, fit-content()).
  • 灵活的尺寸调整功能 (<flex>).

网格尺寸调整算法grid sizing algorithm定义了如何将这些尺寸限制解析为可用轨道尺寸.

11.1. 网格尺寸调整算法

  1. 首先，轨道尺寸算法track sizing algorithm用于解析网格列grid columns的尺寸.

    如果在此步骤中计算网格项的布局取决于块轴block axis中的可用空间available space，则假定具有确定的最大轨道尺寸调整函数max track sizing function的任何行具有该大小并且所有其他行都是无限大的情况下它将具有的可用空间.

    它有助于启发式尝试更准确的初始估计吗？例如。假设它将具有以下最大值的可用空间：
    • 它跨越的所有确定definite轨道大小的总和(如果最小和最大尺寸调整函数都确定，则使用两者中的最大值, 如果确定则为fit-content()的参数).
    • 网格项的最小内容min-content 尺寸, 如果它跨越的任何轨道有 min-content 或 fit-content() 尺寸调整功能.
    • 网格项的自动最小尺寸automatic minimum size, 如果它跨域的任何轨道有 auto 最小尺寸调整函数.
    • 无穷, 如果它跨越的任何轨道有一个 max-content 最小尺寸调整函数 或一个 max-content, auto, 或 <flex> 最大尺寸调整函数.

    这可能会减少必要的重新布局传递的数量，但是在任何情况下它会产生不同的还是更好的结果？我们是否应该将其纳入规范？

  2. 接下来，轨道尺寸算法track sizing algorithm使用上一步中计算的网格列尺寸来解析网格行的尺寸.
  3. 然后，如果任何网格项目的min-content contribution已根据步骤2中计算的行尺寸发生变化，则重复步骤1和2，并使用新的min-content contribution和max-content contribution(仅使用一次).
    如果网格项的内联尺寸inline size取决于网格区域的块尺寸block size，则此周期是必需的. 示例包括包裹列 flex containers (flex-flow: column wrap), orthogonal flows (writing-mode), 和 multi-column elements.
  4. 最后，网格容器grid container的尺寸使用网格的最终尺寸作为其内容尺寸，并且轨道根据align-content和justify-content属性在网格容器内对齐.

    Note: 这可能会在轨道之间引入额外的空间，可能会扩大任何栅格项目的栅格区域，这些栅格区域跨越分配给轨道大小的空间之外的空隙.

一旦确定了每个网格区域的尺寸，网格项目就被布置到它们各自的包含区块中. 为此目的，网格区域的宽度和高度被认为是确定的.

Note: 由于仅使用确定尺寸计算的公式例如stretch fit公式也是确定的，因此被拉伸的栅格项目的尺寸也被认为是确定的.

11.2. 轨道调整术语

min track sizing function
如果轨道尺寸是用minmax()函数确定的，则这是该函数的第一个参数. 如果轨道尺寸是用 <flex> 值 或 fit-content() 函数确定的, 则为auto. 否则, the track’s sizing function.
max track sizing function
如果轨道尺寸是用minmax()函数确定的，则这是该函数的第二个参数. 否则, the track’s sizing function. 在所有情况下, 将 auto 和 fit-content() 视为 max-content, 除非指定 fit-content().
available grid space
独立于每个维度, available grid space 是:

  • 如果网格容器的大小是确定的，则使用生成的内容框的大小.

  • 如果网格容器的大小小于最小内容限制min-content constraint 或最大内容限制 max-content constraint , 则 available grid space 就是这个限制 (并且是确定的).

Note: auto 指定的基于内容尺寸的大小 (如水平书写模式下块级框的高度) 等同于 max-content.

在所有情况下，通过网格容器的最小/最大宽度/高度属性（如果它们是确定的）来限制 available grid space.

free space
等于available grid space减去所有网格轨道(包括gutter)的基本大小的总和，置零. 如果 available grid space 是确定的 indefinite, 则 free space 也是确定的 indefinite.
span count
在可用纬度中网格项横跨网格轨道的数量.

11.3. 轨道尺寸算法

本节的其余部分是轨道尺寸算法，它根据最小和最大曲目尺寸函数计算所使用的轨道尺寸 . 每个轨道都有一个基本尺寸，一个长度在整个算法中增长的<length>，它将最终成为轨道的最终尺寸和一个增长限制， 为基本尺寸提供所需的最大尺寸. 有五个步骤:

  1. 初始化轨道尺寸
  2. 解析内部轨道尺寸

      Why does the infinitely growable flag exist?

      Peter Salas explains:

      Consider the following case:

Two "auto" tracks (i.e. minmax(min-content, max-content) minmax(min-content, max-content)).
Item 1 is in track 1, and has min-content = max-content = 10.
Item 2 spans tracks 1 and 2, and has min-content = 30, max-content = 100.

After resolving min-content/max-content for the first item, we have this.

track 1: base size = 10 growth limit = 10

track 2: base size = 0 growth limit = infinity

Then we resolve min-content/max-content for the second item.

Phase 1 sets the base size of track 2 to 20 so that the two tracks' base sizes sum to 30.
Phase 2 does nothing because there are no relevant tracks.
Phase 3 sets the growth limit of track 2 to 20 so that the two tracks' growth limits sum to 30.
In phase 4, we need to grow the sum of the growth limits by 70 to accommodate item 2.
Two options are:

1. Grow each track’s growth limit equally,
  and end up with growth limits = [45, 55].
2. Grow only the second track’s growth limit,
  and end up with growth limits = [10, 90].

By not considering the just-set growth limit as a constraint during space distribution
(i.e. by treating it as infinity),
we get the second result,
which we considered a better result because the first track remains sized exactly to the first item.
    6. 对于最大内容的最小值 For max-content maximums: 最后继续通过分配额外空间增加具有最大内容的内在最大轨道尺寸函数的轨道的增长限制growth limit,来满足这些网格项的最大尺寸贡献. 但是,通过fit-content()参数限制任何fit-content()轨道的增长.

    对于跨度较大的网格项，重复递增，直到所有网格项都被考虑到.

11.5.1. 跨越跨轨道分配额外空间

To distribute extra space by increasing the affected sizes of a set of tracks as required by a set of intrinsic size contributions,

  1. Maintain separately for each affected base size or growth limit a planned increase, initially set to 0. (This prevents the size increases from becoming order-dependent.)
  2. For each considered item,
    1. Find the space to distribute: Subtract the corresponding size (base size or growth limit) of every spanned track from the item’s size contribution to find the item’s remaining size contribution. (For infinite growth limits, substitute the track’s base size.) This is the space to distribute. Floor it at zero. 
      extra-space = max(0, size-contribution - ∑track-sizes)
    2. Distribute space to base sizes up to growth limits: Find the item-incurred increase for each spanned track with an affected size by distributing the space equally among them, freezing tracks as their size reaches their growth limit (and continuing to grow the unfrozen tracks as needed).

      If a track was marked as infinitely growable for this phase, treat its growth limit as infinite for this calculation (and then unmark it).

      Note: If the affected size was a growth limit, this step has no effect.

    3. Distribute space beyond growth limits: If space remains after all tracks are frozen, unfreeze and continue to distribute space to the item-incurred increase of…
      • when handling min-content or auto base sizes: any affected track that happens to also have an intrinsic max track sizing function; if there are no such tracks, then all affected tracks.
      • when handling max-content base sizes: any affected track that happens to also have a max-content max track sizing function; if there are no such tracks, then all affected tracks.
      • when handling any intrinsic growth limit: all affected tracks.

      For this purpose, fit-content() tracks are treated as max-content until they reach the limit specified as the fit-content() argument, after which they are treated as having a fixed sizing function of that argument.

    4. For each affected track, if the track’s item-incurred increase is larger than the track’s planned increase set the track’s planned increase to that value.
  3. Update the tracks' affected sizes by adding in the planned increase so that the next round of space distribution will account for the increase. (If the affected size is an infinite growth limit, set it to the track’s base size plus the planned increase.)

11.6. 最大化轨道

如果有可用空间 free space, 则将其均匀分配到所有轨道的基本尺寸 base sizes上,并在轨道达到其增长限制growth limits时冻结轨道(如果需要则继续增加未冻结的轨道).

为了这一步的目的: 如果根据最大内容约束max-content constraint来确定网格容器grid container的大小，则可用空间free space是无限的; 如果根据最小内容约束 min-content constraint来确定网格容器的大小, 则可用空间 free space 是0.

如果这步导致网格超过网格容器 grid container的最大宽/高 max-width/height, 则重新执行此步, 当其大小调整为最大宽度/高度时, 将可用网格空间available grid space视为与网格容器的内容盒子大小相等.
